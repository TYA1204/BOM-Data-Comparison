## 叶子节点归组优化 (09:54)

**需求**：方案A — H5F中间父件不输出，仅提取H5F叶子节点合并到P3EM直接上级父组件，P3EM同样只展示叶子数据直接上级。

**实现** (`change_notice.py`):
- 新增叶子判定逻辑：PN ∈ parent_pn_set → 中间节点，否则 → 叶子
- 重写 `group_diffs_by_parent()`: 按叶子的 direct parent_pn 重新归组，不再按原始层级分组
- 移除 self-referencing（所有item均为叶子，组头已显示父件信息）
- H5F叶组通过功能键匹配合入P3EM叶组

**验证结果**: 30条diff → 14个中间节点过滤 → 16个叶子 → 5组, 0违规, 0 H5F中间父件残留, 差异=0

| 分组 | 类型 | 内容 |

| 面壳组件 100P3E MAX | P3EM+H5F合并 | 型材结构件 ADD/DEL |

## BOM感知的父组件名称查找 (10:46)

**问题分析**: 用户发现Word输出中 "液晶组件 100H5F PRO" 组看起来像H5F父组件。经数据库验证，N030106-008401-001 同时存在于 source BOM (bom_id=3) 和 target BOM (bom_id=4)，且两个BOM中名称完全相同（"液晶组件 100H5F PRO 内销 0"）。这是 P3EM BOM 中复用的 H5F 子装配件，不是代码bug。

**确认结论**:
- Group 5 (液晶组件) 是 P3EM 合法父组件，仅有 ADD 项 → 归类为 P3EM 组 ✓
- 名称含"H5F"是因为P3EM BOM直接沿用了H5F的组件命名
- 没有匹配的P3EM"液晶组件"功能键组来接收H5F的DEL数据

**修复**: 虽然此case不是bug，但发现 `get_diff_rows` 中 `parent_lookup` 构建存在隐患——同一PN在源/目标BOM名称不同时可能取错。已修复为按BOM分别查找：
- ADD项 → parent_lookup_tgt (目标BOM)
- DEL项 → parent_lookup_src (源BOM)
- MOD项 → 优先目标，回退源
